본 보고서는 CHEONHONG STUDIO 앱의 전체 소스 코드를 정밀 조사한 결과다. 개인 프로젝트로서 단일 사용자를 대상으로 하며, 일상 루틴 자동화(기상/외출/공부/식사/취침) + 학습 관리 + IoT 연동이라는 고유한 도메인을 갖고 있다.

전반적으로 실전에서 검증된 성숙한 아키텍처를 보여준다. 특히 Firestore dual-write 패턴, 4AM 경계 처리, 3-Layer 캐시, WriteQueue 저널링 등은 일반 개인 프로젝트 수준을 크게 상회한다. 다만 프로젝트 규모가 성장하면서 발생한 구조적 기술 부채와 보안 이슈가 존재한다.

1. 아키텍처 분석

1.1 전체 구조

앱은 크게 4개 레이어로 구성된다:

레이어구성 요소역할

1.2 서비스 의존성 그래프

AppInit (진입점) ├── FirebaseService (Firestore CRUD, 캐시, CF HTTP 프록시) │ ├── LocalCacheService (Hive 기반 로컬 캐시) │ └── FirestoreWriteQueue (중앙 쓰기 큐, Hive 저널) ├── DayService (FSM: idle→awake→outing→studying→returned→sleeping) │ ├── RoutineService (상태 전환, SharedPrefs 저장) │ ├── MealService (식사 독립 추적) │ ├── MovementService (iot.movement Firestore 스트림) │ └── TelegramService (양방향 알림) ├── FocusService (포커스 타이머, ObjectBox 저장, Foreground Task) ├── DoorSensorService (도어센서 Firestore 스트림) ├── WakeService (기상 감지: 수동/센서) ├── SafetyNetService (5분마다 교차 검증 + 알림) ├── FcmService (토큰 등록 + 메시지 핸들링) ├── DataAuditService (일 1회 무결성 검증) └── BackupService (자동/수동 백업)

1.3 데이터 흐름 패턴

핵심 데이터 흐름은 Dual-Write + 3-Layer Cache:

  1. 쓰기: 로컬 캐시 즉시 갱신 → WriteQueue에 enqueue → CF HTTP 전송 → SDK fallback
  2. 읽기: In-memory (5min TTL) → Hive(SharedPrefs) → CF HTTP → Firestore SDK
  3. 스트림: Firestore .snapshots() → write-protected 기간 중 캐시 업데이트 차단

1.4 강점

  • WriteQueue 저널링: Hive에 쓰기 작업 기록 → 앱 크래시 시 재시작 후 자동 복구. 3회 재시도 + 지수 백오프. 실전에서 매우 견고한 패턴.
  • 4AM 경계 일관성: StudyDateUtils.todayKey()가 Dart, Kotlin, JS 전부에서 동일한 4AM 경계 적용. 새벽 활동이 전날에 올바르게 귀속.
  • Completer 기반 in-flight dedup: 동일 문서 동시 읽기 요청을 하나의 Future로 합치는 패턴. 네트워크 효율 극대화.
  • CF HTTP 프록시: gRPC 불안정 대비 REST 경로 확보. SDK fallback 포함.
  • DayState FSM: 상태 기계 설계로 일상 이벤트 순서를 엄격하게 관리.

1.5 구조적 약점

  • God Object 경향: FirebaseService가 3개 part 파일에 걸쳐 약 1,200줄. 읽기/쓰기/스트림/캐시/CF 프록시/rollover 전부 포함.
  • Singleton 남용: 20개 서비스 전부 Singleton. 테스트 어려움, 서비스 간 순환 참조 위험.
  • State Management 미사용: Provider가 pubspec에 있으나 실제 사용 없음. notifyListeners() + 수동 콜백 체인.

2. 버그 및 잠재적 이슈

Critical
B-01: _setNestedValue의 FieldValue.increment toString 파싱이 깨지기 쉬움
lib/services/firebase_history_part.dart:91~113, local_cache_service.dart:142~161

FieldValue.increment()의 로컬 캐시 반영을 value.toString()에서 정규식으로 delta를 추출하는 방식으로 처리. Firebase SDK의 내부 toString 포맷이 변경되면 즉시 실패한다.

// 현재 코드
final str = value.toString();
final match = RegExp(r'increment:\s*([-\d.]+)').firstMatch(str);

개선안: FieldValue.increment를 직접 캐시에 적용하지 말고, delta 값을 별도 파라미터로 전달하는 명시적 API를 사용해야 한다. updateStudyTimeRecord의 effectiveDelta 파라미터처럼.

Critical
B-02: Rollover 중 today doc 날짜 먼저 변경 → 데이터 유실 가능
lib/services/firebase_history_part.dart:464~467

checkDayRollover()에서 아카이빙 전에 today doc의 date를 먼저 새 날짜로 업데이트한다. 이 시점에서 앱이 크래시하면 어제의 today 데이터가 history에 기록되지 않은 채 덮어써진다.

// 아카이빙 전에 날짜 먼저 변경 (위험)
await _db.doc(_todayDoc2).update({
  'date': currentDate,
  '_rolloverInProgress': true,
});

개선안: 아카이빙 완료 후에 date를 변경하거나, 트랜잭션으로 묶어야 한다. _rolloverInProgress 플래그가 있지만, 복구 로직이 stuck 플래그 삭제뿐이라 데이터 복구는 불가.

Warning
B-03: WriteQueue CF 쓰기가 필드별 개별 HTTP → 중간 실패 시 부분 쓰기
lib/services/write_queue_service.dart:183~188

CF HTTP 쓰기는 필드 하나씩 전송한다. 5개 필드 중 3번째에서 실패하면 2개만 쓰여서 문서가 불일치 상태가 된다. SDK fallback이 전체 필드를 다시 쓰지만, CF가 "성공"한 필드는 중복 쓰기.

개선안: CF에 batch write 엔드포인트를 추가하거나, CF 쓰기를 atomic하게 만들어야 한다.

Warning
B-04: 듀얼 문서 동기화 체크에서 isWriteProtected() 타이밍 이슈
lib/services/safety_net_service.dart:482~484

_checkDualDocSync()가 isWriteProtected() 체크로 "최근 쓰기 시만" 실행되지만, write protection이 10초 TTL. 쓰기 직후 5분마다 체크하는 SafetyNet 타이머와 sync가 안 맞아서 대부분의 경우 스킵된다.

개선안: write protection과 무관하게 매 체크마다 실행하되, 수정 실행은 throttle.

Warning
B-05: FocusService._openBox()가 매번 Hive.openBox 호출
lib/services/focus_service.dart:110

_openBox()가 호출될 때마다 Hive.openBox(_boxName)를 실행. Hive는 이미 열린 박스를 반환하지만, 디스크 I/O 오버헤드가 불필요하게 발생할 수 있다.

Future<Box> _openBox() async => Hive.openBox(_boxName);

개선안: Box를 인스턴스 변수로 캐시하고, initialize()에서 한 번만 열어라.

Warning
B-06: Singleton ChangeNotifier의 dispose() 미호출
lib/services/day_service.dart, focus_service.dart, movement_service.dart 등

DayService, FocusService, MovementService 등이 Singleton + ChangeNotifier이지만 dispose()가 앱 수명 동안 절대 호출되지 않는다. Timer, StreamSubscription, Listener가 GC되지 않는다. 단일 사용자 앱이라 큰 문제는 아니지만, RoutineService의 WidgetsBindingObserver가 영원히 등록된 채로 남는다.

_cleanOldFocusCycles()가 _db.doc(_studyDoc).update(keysToDelete)를 직접 호출. WriteQueue를 우회하므로 CF HTTP 경로를 타지 않고, 실패 시 재시도도 없다.

Warning

Warning
F-02: Dual-write 일관성이 eventual consistency에 의존

study doc과 today doc에 동시 쓰기하지만, 트랜잭션이 아니라 enqueue(doc1, fields1); enqueue(doc2, fields2);로 각각 독립 실행. 하나 성공, 하나 실패 시 불일치. SafetyNet과 DataAuditService가 이를 감지/교정하지만, 교정 주기가 5분/24시간.

6.2 캐시 전략 평가

3-Layer 캐시(In-memory → Hive → CF HTTP → SDK)는 잘 설계되어 있다. 특히:

  • Write Protection (10초): 쓰기 직후 스트림/백그라운드 갱신 차단 → optimistic update 보호
  • Background Refresh: 캐시 히트 시 백그라운드에서 서버 최신본 갱신
  • In-flight Dedup: Completer로 동시 요청 병합

다만 persistenceEnabled: false로 Firestore SDK 캐시를 비활성화했으므로, Source.cache 읽기가 항상 실패한다. getArchive()의 캐시 시도가 의미 없다.

결론

CHEONHONG STUDIO는 개인 프로젝트로서 비범한 수준의 엔지니어링을 보여준다. 특히 다음 영역이 인상적이다:

  • Firestore 최적화: CF HTTP 프록시, 3-Layer 캐시, Write Protection, In-flight Dedup, WriteQueue 저널링은 프로덕션 레벨의 안정성을 제공한다.
  • 자동화 파이프라인: 도어센서 → 기상 감지 → 외출 20분 확정 → mmWave 취침 감지 → 전등 자동 제어까지의 end-to-end 자동화가 CF + 앱 + Android 서비스에 걸쳐 유기적으로 동작한다.
  • 데이터 무결성: DataAuditService, SafetyNet 교차 검증, TimeRecord 유효성 검증, dual-write 동기화 체크 등 다층 안전장치.
  • 장애 대비: CF HTTP ↔ SDK fallback, Hive 저널 복구, stuck 플래그 정리, 캐시 자동 무효화.

가장 시급한 개선 사항은 보안(API 키 노출, CF 인증 부재)이다. 기능적 버그는 FieldValue.increment 파싱과 rollover 원자성이 가장 위험하지만, 단일 사용자 환경에서 실제 발현 확률은 낮다.

코드 규모(43K LOC)가 개인 프로젝트치고 상당하므로, Phase 1~2의 즉시 수정 + 점진적 리팩토링 전략이 현실적이다. 전체 재설계는 불필요하며, 현재 아키텍처 위에서 모듈화와 보안 강화에 집중하는 것을 권장한다.
